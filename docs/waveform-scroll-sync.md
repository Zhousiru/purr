# Waveform 滚动同步原理

## 核心问题

为什么 Canvas 内容重绘有延迟，但滚动看起来却是完全同步的？

## 浏览器渲染架构

现代浏览器使用多线程架构：

```
┌────────────────────────────────────────────────────────────────┐
│                        Main Thread                             │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  JavaScript 执行                                          │  │
│  │  DOM 操作                                                 │  │
│  │  样式计算 (Style)                                         │  │
│  │  布局计算 (Layout)                                        │  │
│  │  Canvas 2D API (bindImageData, bindRect, bindText...)     │  │
│  │  React 渲染                                               │  │
│  └──────────────────────────────────────────────────────────┘  │
│                              ↓                                 │
│                         Paint → 生成绘制指令                    │
│                              ↓                                 │
│                        Commit → 提交到合成器                    │
└────────────────────────────────────────────────────────────────┘
                               ↓
┌────────────────────────────────────────────────────────────────┐
│                     Compositor Thread (GPU)                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  图层合成 (Layer Compositing)                             │  │
│  │  CSS transform / opacity 动画                             │  │
│  │  滚动处理 (Scroll Handling)                               │  │
│  │  position: absolute/fixed 元素定位                        │  │
│  └──────────────────────────────────────────────────────────┘  │
│                              ↓                                 │
│                         Draw → 绘制到屏幕                      │
└────────────────────────────────────────────────────────────────┘
```

关键点：
- **Main Thread** 处理 JavaScript、DOM、Canvas 绑制等
- **Compositor Thread** 在 GPU 上运行，处理滚动和图层合成
- 两个线程**并行运行**，Compositor 不会被 Main Thread 阻塞

## 滚动时发生了什么

### 时间线

```
T0: 用户滚动
    │
    ├─→ Compositor Thread: 立即移动所有 absolute 定位的元素
    │                      (Canvas 元素 + VirtualMarks)
    │                      ↓
    │                      屏幕更新 ✓ (用户看到元素移动)
    │
    └─→ Main Thread: 收到 scroll 事件
                     │
                     ├─→ handleScroll() 执行
                     │   - 更新 scrollTop
                     │   - 更新 canvas.style.top
                     │   - 调用 scheduleRender()
                     │
                     └─→ requestAnimationFrame 回调
                         │
                         └─→ render() 执行
                             - clearRect()
                             - bindImageData() × N
                             ↓
                             屏幕更新 ✓ (Canvas 内容更新)

T0 ────────────────────────────────────────────────────→ T1
    │← Compositor 完成 →│← Main Thread 完成 (延迟 1 帧) →│
    │     (~0ms)        │         (~16ms)               │
```

## 缓冲区的作用

如果 Canvas 只有 1 屏高度，快速滚动时会看到白屏：

```
快速滚动 200px (Canvas 高度 = 300px):

Canvas 位置: top = 200 (Compositor 立即更新)
Canvas 内容: 还是 0-300px 的波形 (Main Thread 还没重绘)

用户看到:
┌─────────────────────────────────────────┐
│ 200-300px: 有波形 (旧内容的下半部分)     │
│ 300-500px: 白屏！(超出旧内容范围)        │
└─────────────────────────────────────────┘
```

使用 3 屏缓冲区后：

```
Canvas 高度 = 900px (3 × 300px)
Canvas 内容 = -300px 到 600px 的波形

快速滚动 200px:

Canvas 位置: top = 200 (Compositor 立即更新)
Canvas 内容: 还是 -300px 到 600px 的波形

用户看到:
┌─────────────────────────────────────────┐
│ 200-500px: 有波形 ✓ (在缓冲区范围内)     │
└─────────────────────────────────────────┘
```

## 为什么旧方案不行？

旧方案使用 `position: sticky`：

```html
<div class="container overflow-y-auto">
  <div class="scroll-shim" style="height: totalHeight">
    <VirtualMarks />  <!-- absolute 定位 -->
  </div>
  <canvas class="sticky top-0" />  <!-- sticky 定位 -->
</div>
```

问题：
1. **VirtualMarks** 在 scroll-shim 内，随滚动移动 (Compositor)
2. **Canvas** 是 sticky，固定在视口顶部，不随滚动移动
3. Canvas 内容需要 Main Thread 重绘来反映新的 scrollTop
4. 重绘有延迟 → Canvas 内容和 VirtualMarks 位置不同步

## 新方案为什么有效？

```html
<div class="container overflow-y-auto">
  <div class="scroll-shim relative" style="height: totalHeight">
    <canvas class="absolute" style="top: canvasTop" />
    <VirtualMarks />  <!-- absolute 定位 -->
  </div>
</div>
```

1. **Canvas 和 VirtualMarks** 都在 scroll-shim 内，都是 absolute 定位
2. 滚动时，Compositor Thread 统一移动整个 scroll-shim
3. Canvas **元素位置** 和 VirtualMarks **同步移动** (都是 Compositor)
4. Canvas **内容重绘** 稍后发生 (Main Thread)，但：
   - Canvas 元素已经在正确位置
   - 缓冲区确保旧内容仍然可见
   - 用户感知不到延迟

## 总结

| 组件 | 位置更新 | 内容更新 | 线程 |
|------|----------|----------|------|
| VirtualMarks | 随滚动移动 | N/A | Compositor |
| Canvas 元素 | 随滚动移动 | N/A | Compositor |
| Canvas 内容 | N/A | 重绘 | Main |

**同步的关键**：Canvas 元素的位置移动和 VirtualMarks 都由 Compositor Thread 处理，所以完全同步。Canvas 内容重绘的延迟被缓冲区掩盖，用户感知不到。
